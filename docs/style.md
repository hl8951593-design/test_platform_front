# TestAuto 前端 UI 风格规范

状态：生效
最后核验：2026-06-27

## 1. 规范定位

本规范定义 TestAuto 前端的整体视觉语言和界面实现约束，适用于导航、业务页面、工作台、表单、弹窗、数据列表和运行状态反馈。

- 产品气质：专业、清晰、克制、可信赖的自动化测试工作台。
- 视觉关键词：浅色、蓝色品牌、白色分层表面、紧凑信息密度、明确状态、低干扰动效。
- 使用场景：以桌面端测试研发和项目管理为主，优先保证复杂数据、长表单和多栏工作区的可读性。
- 当前视觉实现以 [`src/styles.css`](../src/styles.css) 为事实来源；本规范用于约束新增和重构界面，不表示项目已经建立 CSS 变量或完整组件库。

新增页面不得自行建立另一套颜色、圆角、阴影或按钮语言。确需新增视觉语义时，应先更新本规范，再落地代码。

## 2. 核心设计原则

1. **内容优先**：装饰服务于层级，不与测试数据、请求参数、运行结果争夺注意力。
2. **状态可信**：运行中、成功、失败、警告和禁用必须来自真实业务状态，并同时使用文字、图标或形状表达，不能只依赖颜色。
3. **层级克制**：主要依靠背景、边框、间距和字重建立层级；阴影轻量使用，渐变只用于品牌和少量重点区域。
4. **操作明确**：一个操作区域通常只有一个主按钮；危险操作必须使用红色语义并经过确认。
5. **密度一致**：总览页适度舒展，编辑器和工作台保持紧凑；同一层级的控件高度、间距和圆角必须一致。
6. **状态完整**：页面和组件应覆盖 loading、empty、error、success、disabled、running/cancelled 等适用状态。

## 4. 字体与排版

- 全局字体：`Inter, "Microsoft YaHei", system-ui, sans-serif`。
- 代码、JSON、路径和技术值：`ui-monospace, SFMono-Regular, Menlo, monospace`。
- 页面标题通常为 `22px`；大区块标题 `24–28px`；面板标题 `15–18px`；正文 `13–14px`；辅助信息 `10–12px`。
- 关键数据可使用 `22–28px` 和 `800–900` 字重；正文不应大面积使用超粗字重。
- 中文正文保持自然字距；仅眉题、短标签和英文大写标识使用较宽字距。
- 正文行高建议 `1.6–1.75`。单行名称可省略，多行说明不得为了对齐而无故截断。

信息层级顺序应为：页面标题 → 区块标题 → 关键值/正文 → 辅助说明/元数据。不要仅通过字号放大制造层级，应同时使用间距和字重。

## 5. 间距、圆角与阴影

### 5.1 间距

使用 `4px` 基础网格，常用值为 `8 / 10 / 12 / 14 / 16 / 18 / 20 / 24 / 28px`。

- 页面区块间距：`18–20px`。
- 面板内边距：常规 `18–24px`，紧凑工作区 `12–16px`。
- 表单字段间距：`12–14px`；标签与控件间距：`6–8px`。
- 操作组间距：`8–10px`。
- 不使用小于 `4px` 的无意义视觉缝隙。

### 5.2 圆角

- 输入框、按钮、分页和小型控件：`8–10px`。
- 列表项、状态块和内部卡片：`10–14px`。
- 页面面板和业务卡片：`16–18px`。
- 大型弹窗：`18–20px`。
- 胶囊标签、进度条和状态徽标：`999px`。

### 5.3 阴影

- 常规卡片使用低透明度阴影，例如 `0 18px 50px rgba(15, 23, 42, .05)`。
- 交互悬停可轻微提高阴影并上移 `1–2px`。
- 弹窗使用更明显但柔和的阴影，例如 `0 30px 90px rgba(15, 23, 42, .22)`。
- 同一容器不要叠加多个强阴影；边框已足以分层时可不使用阴影。

## 6. 页面与布局

### 6.1 应用骨架

- 左侧导航固定/粘滞，展开宽度约 `238px`，收起宽度约 `86px`。
- 顶栏保持粘滞，用于页面标题、项目/环境上下文、搜索和用户入口。
- 主内容区采用浅蓝到白色背景，内容左右留白当前为 `28px`。
- 当前实现是桌面优先，`body` 最小宽度为 `1180px`；不得假定所有复杂工作台已经具备移动端能力。

### 6.2 页面类型

- **总览页**：英雄区/页面摘要 → 指标卡 → 主次内容网格 → 数据列表。
- **列表页**：页面工具栏 → 筛选区 → loading/empty/error 或列表 → 分页。
- **详情页**：返回入口 → 标题与状态 → 主操作 → 分区内容；高风险操作与普通操作分组。
- **编辑器/工作台**：资产或导航栏 → 主画布/编辑区 → 属性或结果栏。各区域优先独立滚动，避免整个页面横向溢出。
- **弹窗**：标题和说明固定在顶部，长内容独立滚动，操作区固定在底部。

### 6.3 TESTAI 工作台

- TESTAI 使用 Codex 式三栏布局：左侧本地 history 和运行控制，中间 `testagnet` 对话区与 composer，右侧 Run/Tool/Approval/Memory/Runbook/Dashboard Inspector 默认收起，并通过中央线程头部的图标按钮展开。
- `/agents` 页面外层应用导航和顶栏遵循项目统一样式；Codex 式视觉只作用于 Agent 内部工作线程、history rail、工具调用活动行和 composer，不改写全局菜单形态。
- `/agents` 工作台在桌面端必须收敛在当前浏览器视口内，不产生页面级纵向滚动；长对话、长 JSON 和左侧历史只在对应内部区域滚动。
- 中央 transcript 只展示用户目标、Agent 回复、思考态、工具调用、审批、迁移、必要的低层事件兜底和结果；`run.*` 生命周期事件只作为状态判断和摘要数据源，不展示为对话卡片；不展示静态系统说明卡，也不展示 dashboard/readiness 权限失败这类全局状态文案。
- 中央 transcript 必须保留当前本地会话内的多轮 turn；继续发送 prompt 时，新一轮用户目标、思考态和 Agent 回复追加到同一线程底部，不得清空或覆盖上一轮问答。
- 中央空状态标题使用“我们应该做什么”；用户发送消息并创建 Run 后，该空状态区域必须消失，避免和用户目标或等待态并存。
- 左右栏为辅助信息，不得挤压中间 transcript；右侧 Inspector 初始收起以优先保留主线程，宽屏可通过详情按钮展开，窄桌面可隐藏右侧详情并保留主线程。
- 中间 transcript 必须以工作线程阅读体验为主：连续 `assistant.delta`、`assistant.message`、`model.delta` 和 `model.message` 合并为一段 Agent 回复并渲染 Markdown 段落、标题、无序列表、有序列表、表格、代码围栏、引用、分隔线、粗体和行内代码，不允许把流式文本片段拆成多条对话卡，也不能把 ` ``` `、`---`、`1.`、`| 表头 |` 这类块级 Markdown 标记裸露成普通文本。
- 工具、命令、执行和输出结果必须按事件位置以轻量内联折叠块展示，折叠态至少露出业务化工具名、状态和关键参数摘要（例如“场景组合”“需求：创建企业场景”），展开态展示 redacted input/output；模型内部 `agent_tool_request` 代码块、包含 `tool_name` 的工具请求 JSON、`model.tool_request_detected` 和 `tool.planned/running/completed` 这类低层过程事件不得作为普通 Agent 回复或事件卡展示；右侧 Inspector 只作为详情补充，不能替代中央线程中的调用痕迹。
- `run.queued`、`run.started`、`run.completed` 等 `run.*` 生命周期事件不进入中央 transcript；低层 `model.*`、`context.*`、`loop.*`、`memory.*` 审计事件不作为普通对话卡展示，避免 start/completed 或 raw payload 把主线程撑成日志面板。
- Composer 的 Enter 键发送必须符合对话工具习惯，Shift+Enter 才换行；发送成功后输入框清空。等待模型或工具返回时，线程中展示带点状动效和耗时计数的“正在思考”轻量活动行，真实回复到达后自动移除。当前 run 有 pending approval 时，composer 上方显示轻量审批确认条，使用警告语义、工具名、权限/风险摘要和“是/否”按钮；该确认条不得展示 approval id、tool call id、hash、snapshot 或 lineage。
- ToolCall、Approval、Migration 和必要的 ContextBuild 在中央时间线中以紧凑折叠块呈现；LoopObservation 属于审计/debug 信息，不作为中央对话卡片展示。完整 JSON、权限、CAS、Memory 证据、LoopObservation 和 runbook 诊断进入右侧详情、timeline/debug、summary 或导出数据；Approval 的底部确认条是当前用户的主决策入口，右侧 Inspector 只作为细节补充。
- 历史列表必须明确本地 MVP 与服务端历史的边界，不能用视觉文案暗示跨设备历史已经可用。
- TESTAI 普通工作区不得把 `run_id`、`conversation_id`、`runtime_snapshot_id`、hash、sequence 等机器标识当作卡片摘要、侧栏字段或 Run Summary 展示；优先使用目标标题、状态、循环进度、工具名、风险原因和可执行动作。
- 历史项的置顶、重命名、导出和删除使用图标按钮时必须提供 `aria-label`；快捷键入口不能替代可见按钮。

所有布局必须允许长名称、长 JSON、错误信息和空数据存在。不得以固定示例内容作为唯一尺寸依据。

## 7. 组件规范

### 7.1 按钮

- 默认高度 `40px`，紧凑操作可使用 `34–36px`；主要点击目标不应低于 `36px`。
- 主按钮：品牌蓝实底白字，一个操作区域通常只保留一个。
- 次按钮：白底中性边框，用于取消、返回和普通操作。
- 弱按钮：浅蓝底或透明背景，用于筛选、查看、低优先级操作。
- 危险按钮：红色文字/边框；最终删除动作可使用红色实底，并必须有确认流程。
- 图标按钮必须提供 `aria-label` 或等价可访问名称，不能只靠图标猜测含义。
- disabled 和 loading 均保留原按钮宽度；loading 使用真实执行状态，不伪造进度。

### 7.2 表单

- 输入框、选择器和文本域使用白色表面、`#dbe3ef` 边框和 `10px` 左右圆角。
- 聚焦态使用品牌边框加浅色 focus ring，不能只移除浏览器轮廓。
- 标签位于控件上方；必填标识、帮助文本和错误文本各自保持固定位置。
- 校验失败使用红色边框、浅红背景和明确错误文案；不要只弹 toast。
- JSON、请求体、响应体和表达式使用等宽字体，并保留原始值类型。

### 7.3 卡片、面板与表格

- 主卡片使用白底、细边框、`16–18px` 圆角和轻阴影。
- 嵌套区域优先用次表面或分隔线，不要无限套卡片和阴影。
- 表头使用次表面和辅助文字；行间以浅边框分隔。
- 行操作靠右并保持稳定位置；运行中可用左侧品牌色条辅助识别。
- 数字、状态和操作列尽量保持短而稳定，名称和描述列承担弹性宽度。
- 属性检查器中的低频复杂配置默认折叠，折叠标题应保留名称、摘要和明确的展开状态；当前步骤的紧凑执行操作放在检查器标题栏右侧。执行完成后在检查器内保留状态码、耗时和数据数量摘要，完整响应由用户点击摘要后在任务型弹窗中查看。

### 7.4 状态、反馈与空状态

- 状态徽标采用胶囊形，文字与语义色同时出现。
- loading：使用旋转图标、骨架或明确文案，保留原布局，避免内容跳动。
- empty：说明“为什么为空”并在可恢复时给出下一步操作。
- error：就近展示错误原因和重试入口；全局 toast 只作为补充。
- success：确认动作结果，但不要长期占据主界面。
- cancelled/interrupted：使用独立文字说明，不能伪装为失败或成功。

### 7.5 弹窗与确认

- 遮罩覆盖完整视口；弹窗宽度按任务复杂度选择，不以“大”代替清晰分组。
- 标题说明操作对象，正文说明影响，主操作与取消操作置于底部右侧。
- 删除确认必须显示对象名称或影响范围；忙碌期间禁用重复提交和关闭风险入口。
- Agent approval 确认使用贴近 composer 的底部确认条而非全屏遮罩，按钮文案保持“是/否”，同时用 `aria-label` 表达完整动作；忙碌期间两个决策按钮都禁用。
- AI 生成类弹窗必须区分“生成草稿”和“确认保存”两个动作；warnings 使用警告语义就近展示，任何会真实执行候选用例或调用业务接口的开关都必须默认关闭并要求二次确认。
- AI Run 执行面板必须把模型输出、步骤事件和工具调用合并为单一流式输出，并在新返回到达时自动展示最新内容；不再为工具调用单独拆分调用历史区域。
- AI 生成结果预览中的节点数据优先用名称、路径、目标字段、期望值等结构化短行展示；原始复杂 JSON 只放在可展开的 config 详情里兜底。
- AI 扩展接口用例返回的草稿必须复用测试用例编辑弹窗的结构化控件：方法与路径置顶，`Params`、`Headers`、`Body`、`断言` 通过标签页和 Key/Value 行编辑；不要把候选用例整体作为 JSON 让用户直接修改。
- AI 扩展接口用例运行中必须复用 AI Run 流式执行面板展示 `model.delta`、`run.*`、`step.*` 和 `tool.*`，不能退化成只有“正在扩展”的静态等待卡。
- 场景画布连线只用于主测试用例节点之间；前置/后置动作和节点内取值、引用、工具明细使用主测试用例卡片内的分组摘要、缩进卡片或折叠详情表达，不画内部连线。前置/后置动作分组默认收起，分组标题保留数量和状态摘要。
- 弹窗应支持键盘焦点、`Escape`（无不可逆进行中操作时）和合理的焦点回归。

## 8. 图标、图像与数据可视化

- 统一使用 Material Symbols，常规尺寸 `17–20px`，空状态和品牌图标可放大。
- 相同语义使用相同图标；禁止在同一操作中混用不同图标风格。
- 图标用于增强识别，不替代关键操作文字和状态文案。
- 产品截图、附件和头像必须定义裁切方式、空状态、加载失败状态及替代文本。
- 富文本中粘贴的截图应出现在当前光标位置，并按正文宽度等比缩放；独立附件区只展示通过附件选择入口添加的图片，避免同一图片重复出现。详情正文图片使用放大光标，双击后进入深色灯箱；多图提供左右切换、序号、文件名、关闭按钮和键盘方向键/Esc 操作。
- 图表以品牌蓝为主，语义色为辅；必须提供标题、单位、图例或可读数值，不能仅靠颜色区分类别。

## 9. 交互、动效与可访问性

- 常规过渡为 `160–200ms ease`，仅用于颜色、边框、阴影、轻微位移和展开收起。
- 禁止使用动效模拟后端运行进度或步骤完成。
- 支持 `prefers-reduced-motion: reduce`，此时关闭非必要过渡和位移。
- 所有可交互元素必须有 hover、focus-visible、active 和 disabled 中适用的状态。
- 文本与背景保持足够对比度；语义不能仅依赖颜色。
- 键盘可以到达主要操作、标签页、菜单、列表选择和弹窗操作；焦点轮廓清晰可见。
- 列表行内的浮层菜单打开时，触发行必须提升到相邻行之上，菜单不得被后续卡片、分页或容器边界遮挡。
- 触发器、输入和错误提示应使用正确语义元素及必要的 ARIA 关联。

## 10. 响应式与密度

- 首要验收视口为桌面端；复杂三栏工作台在窄桌面下优先压缩辅助区和次要文案，不挤压主编辑区到不可用。
- 同一工作台的左右辅助面板承担对等层级时应保持等宽，主编辑区使用剩余弹性宽度。
- `1600px` 以下可隐藏非关键说明、收紧导航与顶栏；`1300px` 左右允许多栏内容转为单列或纵向堆叠。
- 简单列表和分页在更窄视口可换行，但不得制造横向页面滚动。
- 若新增真正的移动端能力，必须单独定义导航、表格、弹窗和工作台降级策略，并更新本规范；当前不得宣称全站移动端适配完成。

## 11. 文案风格

- 使用简短、明确、可执行的中文；按钮优先使用“动词 + 对象”，如“新建场景”“保存配置”“重新加载”。
- 状态文案描述事实，如“运行中”“已取消”“加载失败”，避免“好像失败了”等模糊表达。
- 空状态先说明原因，再提供下一步；错误文案包含用户可采取的动作。
- 场景前置/后置动作入口必须贴近所属测试用例卡展示，并在选择弹窗中重复显示绑定用例；禁止使用占据画布的大型全局前置/主流程/后置分区表达节点私有动作。节点内动作使用主测试用例卡片底部的“前置动作/后置动作”分组摘要，不把每个动作都铺成同等重量的大卡片，也不在主卡片外形成独立宽分区。
- 内置动作编辑器使用一致的图标标题、简短用途说明和结构化字段；脚本必须使用带行号、语法高亮、括号匹配和焦点态的代码编辑器，不得退化为普通多行文本框。代码补全项采用“名称 + 右侧中文用途说明”，并只展示沙箱实际允许的变量、函数和关键字；代码容量、沙箱禁用能力和变量错误在编辑器附近就地展示；JSON 输入也必须就地展示格式错误。
- 同一业务概念必须遵循项目统一术语，不能为视觉简洁擅自改写字段含义。

## 12. 实现约束

- 优先复用 `src/components/` 的公共组件和 `src/styles.css` 的现有模式，避免页面内复制按钮、鉴权提示、响应解包或状态映射。
- 新样式使用可表达业务/组件角色的类名，不以颜色或具体坐标命名，例如使用 `.run-status` 而不是 `.blue-box`。
- 不在 JSX 中散落可复用视觉值；动态尺寸或进度值可使用行内样式，固定视觉规则进入样式文件。
- 当前色值仍大量直接写在 `src/styles.css` 中。新增代码应复用本规范列出的现有值，避免继续扩张近似色；若进行 token 化改造，应作为独立变更并同步本文。
- 视觉调整不得改变接口语义、数据类型、权限边界或真实运行状态。

## 13. UI 交付检查清单

新增或修改页面交付前，至少检查：

- [ ] 页面结构符合对应页面类型，主操作唯一且清晰。
- [ ] 颜色、字号、圆角、间距、阴影复用现有视觉语言。
- [ ] loading、empty、error、success、disabled 和中断状态按适用范围实现。
- [ ] hover、focus-visible、键盘操作和 reduced-motion 行为可用。
- [ ] 长名称、长错误、长 JSON、无数据和高数据量不会破坏布局。
- [ ] 语义状态同时有文字/图标或形状，不只依赖颜色。
- [ ] 复杂页面和样式相关修改至少在 `1920×1080` 桌面视口截图核验正常；必要时补充窄桌面视口核验。
- [ ] UI 行为变化已同步模块架构文档和 `docs/frontend-feature-logic.md`。
- [ ] 新增视觉模式或规范变化已同步本文和 `docs/README.md`。
- [ ] 已运行 `npm run docs:check`、`npm test -- --run` 和 `npm run build`。
