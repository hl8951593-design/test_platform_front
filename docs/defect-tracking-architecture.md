# 缺陷跟踪前端架构

状态：当前实现
最后核验：2026-06-18

本文档描述缺陷模块当前前端实现。页面入口为 `/defects`，主文件为 `src/pages/DefectsPage.tsx`，接口适配层为 `src/api/defects.ts`。

## 模块定位

缺陷模块用于记录和跟踪 Bug，不参与测试执行调度，也不模拟执行结果。缺陷归属于当前项目，前端必须显式携带 `project_id`，后端负责校验项目归属、用户权限和状态流转合法性。

## 状态流程

缺陷状态枚举如下：

| 前端值 | 展示文案 | 说明 |
| --- | --- | --- |
| `new` | 新创建 | 新记录的 Bug，尚未进入处理 |
| `active` | 已激活 | 已进入处理队列 |
| `confirmed` | 已确认 | 已确认可复现或需要修复 |
| `fixed` | 已修复 | 开发已修复，等待验证 |
| `verified` | 已验证 | 测试已验证修复结果 |
| `closed` | 已关闭 | 缺陷生命周期结束 |
| `reopened` | 重新激活 | 关闭或验证后再次发现问题 |

页面提供流程按钮进行常用流转：

| 当前状态 | 可流转到 |
| --- | --- |
| 新创建 | 已激活、已确认、已关闭 |
| 已激活 | 已确认、已修复、已关闭 |
| 已确认 | 已修复、已关闭 |
| 已修复 | 已验证、重新激活 |
| 已验证 | 已关闭、重新激活 |
| 已关闭 | 重新激活 |
| 重新激活 | 已激活、已确认、已修复 |

前端按钮只表达推荐流程，不作为最终权限或合法性判断。后端仍应在状态接口中校验不允许的跳转。

## 数据模型

前端标准模型为 `Defect`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `string` | 后端缺陷主键 |
| `projectId` | `number` | 项目边界 |
| `title` | `string` | Bug 标题 |
| `assignee` | `string` | 指派人展示名或账号 |
| `type` | `DefectType` | Bug 类型 |
| `urgency` | `DefectUrgency` | 紧急程度 |
| `status` | `DefectStatus` | 当前流程状态 |
| `contentHtml` | `string` | 富文本内容 HTML |
| `attachments` | `DefectAttachment[]` | 图片附件元数据和当前短期访问地址 |
| `reporter` | `string` | 报告人 |
| `createdAt` / `updatedAt` | `string` | 创建和更新时间 |

`src/api/defects.ts` 负责把后端 `bug_type`、`content_html`、`assignee_name`、`reporter_name` 等 snake_case 字段映射成页面使用的 camelCase 模型。

## 页面能力

| 能力 | 当前实现 |
| --- | --- |
| 缺陷列表 | 摘要行只展示标题、状态、紧急程度、类型、更新时间、指派人和提出人；支持搜索、筛选和分页 |
| 缺陷详情 | 点击摘要进入 `/defects/{id}`，通过单条详情接口读取正文、附件和最新元数据 |
| 新建缺陷 | 维护标题、指派人、Bug 类型、紧急程度、状态、富文本内容和图片附件；标题和内容使用红色必填标记 |
| 编辑缺陷 | 仅从详情页进入；附件未改动时省略 `media_ids`，改动时提交完整 ID 数组 |
| 状态流转 | 列表使用三点菜单收纳可用状态；菜单打开时提升当前卡片层级，避免被相邻记录遮挡；详情页同时提供明确流转入口 |
| 删除缺陷 | 仅在详情页提供，二次确认后调用删除接口 |
| 富文本编辑 | 使用 `contentEditable`，支持加粗、斜体和列表；粘贴图片上传后插入当前光标位置，正文持久化媒体 ID 占位地址 |
| 正文图片预览 | 详情页双击正文图片打开灯箱；按正文顺序组织图片，支持左右按钮和方向键循环切换，Esc、关闭按钮或点击遮罩退出 |
| 图片附件 | “选择图片”添加独立附件；粘贴图片属于正文内嵌媒体，不在附件区重复展示；两者均调用 `/media/images` 并通过 `media_ids` 绑定 |

富文本内容在提交和列表预览前会移除 `script`、`style`、`iframe`、事件属性和 `javascript:` 地址。后端仍应进行服务端清洗和存储层安全控制。

预签名 URL 是临时访问凭证，不能写入 `content_html`。正文内嵌图片使用 `<img src="/__defect_media__/12" data-media-id="12" alt="checkout.png">` 作为稳定引用；相对 `src` 是跨服务端 HTML 清洗的主标记，`data-media-id` 是前端编辑辅助属性。编辑和详情展示时按 `attachments` 中同 ID 的当前 `download_url` 替换占位地址，加载失败后调用刷新地址接口。对已经被清洗成无 `src`、仅保留 `alt` 的正文图片，前端按文件名和出现顺序匹配附件并恢复显示，保存后升级为新标记。旧的无图片 Bug、独立附件和历史富文本内容继续兼容。

编辑附件遵循后端三态语义：未改动时不传 `media_ids` 并保留原附件；改动后传完整 ID 数组替换绑定；删除全部后传空数组。从正文删除内嵌图片会在保存时解绑对应媒体。选择图片和粘贴图片均按文件逐个使用 `multipart/form-data` 上传，字段名为 `file`。

## 页面状态

- 未选择项目时，页面提示先选择项目，并禁用新建缺陷。
- 加载中展示列表 loading 状态。
- 接口失败展示错误空态。
- 无数据或筛选无结果展示 empty 状态。
- 新建/编辑弹窗采用固定头部和底部操作区，中间表单区域滚动，避免富文本区域挤压或遮挡保存按钮。
- 保存、删除和流转过程中禁用对应操作，避免重复提交。
- 保存成功后关闭弹窗并重新读取后端列表。
- 浏览器前进、后退和直接刷新 `/defects/{id}` 时保持详情路由，并重新读取后端详情。
