# 文档治理与 AI 开发规范

状态：生效
最后核验：2026-06-30

## 1. 目标

项目文档必须让开发者和 AI 能回答四个问题：

1. 当前系统已经实现了什么。
2. 数据、状态和权限由谁负责。
3. 前后端通过什么稳定契约协作。
4. 下一步工作是什么，哪些内容仍是目标设计而非当前事实。

文档不是代码完成后的补充说明。涉及架构、数据模型或接口的变更，文档与测试是同一项交付的一部分。

## 2. 文档分层与权威来源

| 层级 | 文档 | 负责内容 |
| --- | --- | --- |
| 仓库入口 | `README.md`、`AGENTS.md`、`docs/README.md` | 项目启动、AI 阅读顺序、文档导航 |
| 全局架构 | `docs/technical-architecture.md` | 模块边界、数据归属、权限、跨模块关系 |
| 当前行为 | `docs/frontend-feature-logic.md`、模块架构文档 | 页面能力、交互、状态优先级、前端责任 |
| 接口契约 | `api_docs/api_*.md`、`docs/*-contract.md` | 路径、字段、状态机、兼容和错误语义 |
| 交付计划 | `docs/frontend-development-plan.md` | 已完成、质量基线、风险和优先级 |
| 可执行证据 | TypeScript 类型、API 映射、自动化测试 | 验证文档是否与实现一致 |

## 2.1 文档收敛与合并规则

当前前端仓库保留了后端 API 镜像、前端模块架构和早期专项方案。为了避免同一事实在多个文件分叉，后续按以下规则维护：

| 内容 | 当前权威文档 | 处理规则 |
| --- | --- | --- |
| 前端整体架构、路由、权限和数据归属 | `docs/technical-architecture.md` | 只记录当前实现，不写后端尚未落地的目标能力 |
| 页面行为和交互细节 | `docs/frontend-feature-logic.md` 与模块架构文档 | 页面代码变化优先更新这里 |
| 开发状态、剩余计划和质量基线 | `docs/frontend-development-plan.md` | 专项计划完成后合并到该主计划，旧计划改为跳转 |
| Agent 工作台当前实现 | `docs/agent-runtime-frontend-architecture.md` | 取代旧 `agent-codex-prototype` 方案文档 |
| 后端 API 契约镜像 | `api_docs/api_*.md` | 只用于前端联调查阅，权威源仍是后端仓库 `docs/api_*.md` |
| 历史方案或一次性原型 | 保留短入口并指向当前权威文档 | 不继续复制完整方案，避免过期 |

当两个文档描述同一事实时，保留一个当前实现文档，另一个改为“已合并/历史入口”。只有路径、字段、状态机这类后端契约才进入 `api_docs/`；前端布局、组件、交互和计划不得继续新增到 `api_docs/`。

出现冲突时不能默默选择其中一个版本：

1. 先通过类型、实现和测试确认“当前事实”。
2. 再确认 contract 是否表达“目标契约”。
3. 当前事实与目标契约不一致时，在开发计划记录缺口，并在 contract 标明状态。
4. 修复完成后同时更新代码、测试、模块文档和接口契约。

## 3. 文档状态

关键技术文档应在标题后标明：

- `状态：当前实现`：描述已经落地并有代码或测试证据的行为。
- `状态：目标契约`：要求前后端共同实现，但可能仍有计划项。
- `状态：草案`：尚未确认，不能作为实现依据。
- `最后核验：YYYY-MM-DD`：最近一次对照代码和测试的日期。

不要使用“后续支持”“当前已经”等模糊表述掩盖状态差异。未完成项必须进入开发计划。

## 4. 统一术语

| 中文 | 英文/字段 | 定义 |
| --- | --- | --- |
| 场景 | Scenario | 可版本化的多步骤自动化流程定义 |
| 数据集 | Dataset | 一组共享变量和多条测试记录的容器 |
| 测试记录 | Dataset Record / `records[]` | 一次完整、独立的场景运行输入 |
| 请求覆盖 | Request Override / `request_overrides[]` | 对指定步骤 Path、Header、Query 或 Body 字段的运行时覆盖 |
| 调试记录 | Scenario Run | 后端持久化的整场运行摘要或详情 |
| 单步调试 | Step Debug | 未保存步骤配置的一次临时执行，不等同于整场运行 |
| 响应取值 | Extraction | 从步骤响应中提取并命名的值 |
| 上游引用 | Binding | 把上游提取值写入下游请求字段 |

禁止把“测试记录”写成一个字段的多个离散值。多个字段在同一条记录中共同组成一次测试输入。

## 5. 变更时的同步要求

| 改动 | 同步动作 |
| --- | --- |
| 新增页面入口或导航 | 更新功能逻辑、路由表和文档索引 |
| 修改数据模型 | 更新 TypeScript 类型、序列化、contract、示例和兼容策略 |
| 修改执行状态 | 更新状态机、视觉语义、状态优先级和测试 |
| 修改 API | 更新路径、方法、请求、响应、错误、权限和幂等说明 |
| 新增兼容读取 | 写明兼容输入、标准输出和移除条件 |
| 完成功能 | 从计划移入“已完成”，更新核验日期和测试证据 |
| 发现未实现设计 | 标为目标契约并加入计划，禁止描述成当前能力 |

## 6. AI 开发流程

AI 开始任务前：

1. 读取 `AGENTS.md`、`docs/README.md` 和对应模块文档。
2. 检查工作区已有改动，不覆盖未提交内容。
3. 从类型和 API 适配层确认数据模型，从测试确认关键行为。
4. 搜索旧字段名、旧状态和同义术语，评估影响范围。

AI 完成任务前：

1. 更新受影响的模块文档、contract 和开发计划。
2. 检查示例 JSON 与实际序列化结构一致。
3. 运行文档检查、测试和构建。
4. 在交付说明中明确已完成内容、验证结果和仍存在的目标契约。

## 7. 自动检查

运行：

```bash
npm run docs:check
```

检查内容包括：

- Markdown 本地链接是否存在。
- 核心文档是否进入索引。
- 场景 API 文档是否包含异步执行和记录化数据驱动关键契约。

自动检查不能替代语义审查。每次模型、接口或状态机变化仍需人工或 AI 对照源码核验。
