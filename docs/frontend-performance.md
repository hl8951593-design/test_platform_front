# 前端性能优化文档

状态：当前规范
最后核验：2026-06-15

本文档记录 TestAuto 前端性能相关的约定和当前已完成的优化，后续新增页面、列表、弹窗和接口调用时应同步遵守。

## 优化目标

| 目标 | 说明 |
| --- | --- |
| 避免重复渲染 | 稳定传给子组件的函数和数据，减少无意义刷新 |
| 避免过期请求污染状态 | 项目、环境、用例等请求在快速切换时不能用旧响应覆盖新状态 |
| 控制列表成本 | 列表空态、加载态和数据态清晰拆分，后续大数据量时易接分页或虚拟列表 |
| 降低表单维护成本 | 新增/编辑弹窗不预置 mock 数据，避免无效字段参与保存和调试 |
| 保持接口边界清晰 | 页面组件只调用 `src/api/` 封装，不直接拼接请求 |

## 当前已完成

### 全局壳层

| 文件 | 优化 |
| --- | --- |
| `src/App.tsx` | `navigate`、`onAction`、`handleProjectCreated`、菜单折叠函数使用稳定回调 |
| `src/App.tsx` | `Navigation`、`TopBar`、`Toast` 使用 `memo`，减少全局状态变化导致的重复渲染 |
| `src/App.tsx` | 项目列表、环境列表请求增加过期响应保护，快速切换时旧响应不会覆盖新状态 |
| `src/App.tsx` | 顶部环境下拉跟随项目加载真实环境，不再使用静态环境 mock |

### 接口测试用例

| 文件 | 优化 |
| --- | --- |
| `src/pages/ApiPage.tsx` | 测试用例列表加载函数使用稳定回调 |

## 后续开发规则

### 状态与渲染

1. 传给深层组件、列表行、弹窗的函数，如果会导致明显重渲染，应使用稳定回调。
2. 由 props 或 state 派生出的布尔值、列表过滤结果、映射结果，如果计算成本会随数据量增长，应使用派生缓存。
3. 不要在 JSX 中直接写大段映射、格式化、解析逻辑；复杂转换应放到组件外部纯函数或 API 映射层。
4. 页面级组件可以管理业务状态，通用 UI 组件应尽量保持无业务副作用。

### 请求与数据一致性

1. 依赖项目、环境、筛选条件的请求，必须处理过期响应。
2. 任何列表接口都不应依赖前端默认 ID；项目 ID、环境 ID 必须来自当前选择或后端返回。
3. 页面组件只调用 `src/api/` 中的函数，不直接写 `fetch`。
4. API 封装负责鉴权头、错误消息、统一响应解包和字段映射。

### 列表与表格

1. 列表必须区分加载中、加载失败、空数据、正常数据四类状态。
2. 大列表后续接入后端分页，避免一次性渲染全部数据。
3. 单页超过 200 行可见数据时，优先考虑分页；超过 1000 行时，再考虑虚拟滚动。
4. 表格行按钮必须阻止行点击冒泡，避免运行、编辑等操作互相触发。

### 表单与弹窗

1. 新增表单不得预置业务 mock 数据；只允许展示 placeholder 或空态提示。
2. 编辑表单只回填后端真实返回的数据。
3. 保存前必须校验项目、环境、必要字段和 JSON 格式。
4. 调试未保存内容时，必须使用用户当前编辑内容生成请求体。

## 检查清单

新增页面或模块时，合并前至少确认：

| 检查项 | 结果 |
| --- | --- |
| 是否存在接口请求竞态覆盖风险 | 必须处理 |
| 是否存在默认项目/默认环境兜底 | 不允许 |
| 是否存在新增表单 mock 数据 | 不允许 |
| 是否区分加载/错误/空态 | 必须区分 |
| 是否通过 `npm test -- --run` | 必须通过 |
| 是否通过 `npm run build` | 必须通过 |
